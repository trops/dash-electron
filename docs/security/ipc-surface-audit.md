### widget-passthru — verdicts

All four channels were manually reviewed (cf. `test(security):
regression-pin widget-scoped storage in scheduler + notification
controllers` in dash-core 0.1.505). The audit's `widget-passthru` flag
fired because none of the four bodies references a known gate name —
correct, because the scheduler and notification controllers scope by
**Map keys / filter on `task.widgetId === widgetId`** rather than via a
named gate. That is a valid scoping pattern; the controllers are
already widget-keyed.

| Channel                        | Delegate            | Scoping mechanism                                                                                    | Verdict                                                                                   |
| ------------------------------ | ------------------- | ---------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `NOTIFICATION_SET_PREFERENCES` | `setPreferences`    | `electron-store` `instances[widgetId]` keyed write; merge into existing `instances[widgetId]` only   | OK — pinned in `notificationController.test.js`                                           |
| `SCHEDULER_GET_TASKS`          | `getTasks`          | Map iteration filtered by `task.widgetId === widgetId`                                               | OK — pinned in `schedulerController.test.js` (cross-widget read returns nothing)          |
| `SCHEDULER_REMOVE_TASKS`       | `removeTasks`       | Map iteration filtered by `task.widgetId === widgetId`; also calls `pendingResults.delete(widgetId)` | OK — pinned in `schedulerController.test.js` (cross-widget delete leaves B's tasks alone) |
| `SCHEDULER_GET_PENDING`        | `getPendingResults` | `pendingResults.get(widgetId)` keyed read; returns `[]` for unknown widgetId                         | OK — partial pin (empty-state contract); seeding requires running `fireTask`              |

**Trust model note.** The four channels — like the existing MCP and fs
gates — trust the renderer-supplied `widgetId` at face value. A widget
can pass another widget's `widgetId` and read or remove that widget's
tasks. This is a system-wide design property, not a per-channel gap:
all renderer-supplied `widgetId` values are trusted across the gate
surface today. Hardening the trust model (e.g. tying `widgetId` to
`webContents.id` via a registry) is a separate, much larger slice and
would affect every per-widget gate at once.

### dead — cleanup status

The original baseline classified seven channels as dead. On manual review:

-   **Six were truly dead and have been removed in dash-core** (see
    `chore(security): remove 6 dead widget-facing IPC surfaces`):
    -   `ALGOLIA_SAVE_SYNONYMS` (algoliaApi.saveSynonyms — was invoking
        `undefined` because the constant itself was never declared)
    -   `CLIENT_CACHE_INVALIDATE`, `CLIENT_CACHE_INVALIDATE_ALL`,
        `RESPONSE_CACHE_CLEAR`, `RESPONSE_CACHE_STATS`
        (clientCacheApi — entire file deleted; dash-electron's preload
        had been overriding the namespace anyway, but its handlers are
        also unwired — see follow-up below)
    -   `LAYOUT_SAVE` (layoutApi.saveLayoutForApplication +
        layoutController.saveLayoutForApplication)
-   **One was a false positive**: `THEME_LIST`. The audit script's
    parser only matches ALL_CAPS constant names in
    `loggedHandle(NAME, …)`. The actual handler in
    `dash-electron/public/electron.js` line 986 uses an inline string
    `loggedHandle("theme-list", (e, message) => …)`. The channel IS
    wired and has live callers in `dash-core/src/Api/ThemeApi.js` and
    `dash-core/src/Api/ElectronDashboardApi.ts`. **Kept.**

After this slice the audit's dead-pile drops to **0** for the named-
constant audit. The script will continue to flag THEME_LIST as `dead`
on re-run until the parser learns to detect inline-string handlers
(separate slice, deferred).

### Follow-up: dash-electron's own dead clientCache + responseCache

`dash-electron/public/preload.js` exposes `mainApi.clientCache.{invalidate,
invalidateAll}` and `mainApi.responseCache.{clear, stats}` via inline
strings (`"client-cache-invalidate"` etc.), but no main-process handler
exists for any of them. The `invoke` calls go nowhere. Same pattern as
the dash-core methods we just removed; the cleanup is symmetric but
lives in dash-electron and is independent of this slice.

### Recommended next slices

1. **Cleanup dash-electron's own dead namespaces** — apply the same
   pattern to the preload-exposed `mainApi.clientCache` and
   `mainApi.responseCache` whose `invoke` channels have no
   main-process handler.
2. **Network domain JIT** — the original Phase 3 plan: gate
   `READ_DATA_URL`, the `WS_*` family, and `THEME_EXTRACT_FROM_URL` with
   a `domains.network` grant + JIT consent. Architecturally identical to
   Phase 2; the heaviest part is the modal's hostname-allowlist UI.
3. **Improve audit script's parser** — detect inline-string handlers
   (`loggedHandle("foo", …)`) so `THEME_LIST` stops being a false
   positive, plus surface inline-string `invoke()` calls so
   dash-electron's own preload-side dead namespaces are flagged
   automatically. Unlocks putting the audit in CI.
4. **Domain-neutral rename** — `enforceWidgetMcpPermissions` →
   `enforceWidgetPermissions`, `enableJitConsent` stays. Mechanical, but
   touches every read-flag site; defer until the next big slice settles.
5. **Tighten the widgetId trust model** (large) — today every gate
   trusts the renderer-supplied `widgetId` at face value. A `widgetId`
   ↔ `webContents.id` registry would harden every gate at once. Out of
   scope until the rest of the campaign settles.

### Heuristic limitations (known, accepted)

-   The delegate-extractor takes the first identifier-shaped call after
    `getSenderWindow(e)`. Multi-step dispatchers (e.g. a switch that
    routes by message type) won't be picked up; those handlers will
    classify as `system` even when the routed-to function gates. Fix
    requires real AST traversal — out of scope for this slice.
-   A "gated" classification requires the delegate function name to
    appear in a controller file with a gate ref within ~80 lines of its
    declaration. Refactors that move the gate into a helper called more
    than 80 lines later will silently downgrade the classification.
-   Inline string-literal channels (`ipcRenderer.invoke("popout-open")`)
    are intentionally skipped — they're never widget-facing in this
    codebase and adding them noises the table.
