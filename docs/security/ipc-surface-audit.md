### widget-passthru — verdicts

All four channels were manually reviewed (cf. `test(security):
regression-pin widget-scoped storage in scheduler + notification
controllers` in dash-core 0.1.505). The audit's `widget-passthru` flag
fired because none of the four bodies references a known gate name —
correct, because the scheduler and notification controllers scope by
**Map keys / filter on `task.widgetId === widgetId`** rather than via a
named gate. That is a valid scoping pattern; the controllers are
already widget-keyed.

| Channel                        | Delegate            | Scoping mechanism                                                                                    | Verdict                                                                                   |
| ------------------------------ | ------------------- | ---------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| `NOTIFICATION_SET_PREFERENCES` | `setPreferences`    | `electron-store` `instances[widgetId]` keyed write; merge into existing `instances[widgetId]` only   | OK — pinned in `notificationController.test.js`                                           |
| `SCHEDULER_GET_TASKS`          | `getTasks`          | Map iteration filtered by `task.widgetId === widgetId`                                               | OK — pinned in `schedulerController.test.js` (cross-widget read returns nothing)          |
| `SCHEDULER_REMOVE_TASKS`       | `removeTasks`       | Map iteration filtered by `task.widgetId === widgetId`; also calls `pendingResults.delete(widgetId)` | OK — pinned in `schedulerController.test.js` (cross-widget delete leaves B's tasks alone) |
| `SCHEDULER_GET_PENDING`        | `getPendingResults` | `pendingResults.get(widgetId)` keyed read; returns `[]` for unknown widgetId                         | OK — partial pin (empty-state contract); seeding requires running `fireTask`              |

**Trust model note.** The four channels — like the existing MCP and fs
gates — trust the renderer-supplied `widgetId` at face value. A widget
can pass another widget's `widgetId` and read or remove that widget's
tasks. This is a system-wide design property, not a per-channel gap:
all renderer-supplied `widgetId` values are trusted across the gate
surface today. Hardening the trust model (e.g. tying `widgetId` to
`webContents.id` via a registry) is a separate, much larger slice and
would affect every per-widget gate at once.

### widget-passthru — dash-electron preload verdicts

The 7 channels surfaced once the parser learned to read inline-string

on `mainApi` to every renderer context — including widget code — and
none of their handlers verify the caller's identity. Verdicts:

| Channel                    | Risk                 | Verdict                                                                                                                   |
| -------------------------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `widget-mcp:set-grant`     | ⚠️ Hardened (DiD)    | Broadening calls now require an OS-native confirmation (renderer-untamperable). See **Native-dialog hardening** below.    |
| `widget-mcp:revoke`        | Medium (DoS)         | A widget can wipe another widget's grant. Real gap; same root cause.                                                      |
| `widget-mcp:revoke-server` | Medium (DoS)         | Partial-revoke variant. Real gap; same root cause.                                                                        |

| `widget-popout-open`       | Low (UX abuse)       | Open a popout for any widget+workspace. Could spam popouts; not a security gap per se.                                    |
| `widget-popout-set-title`  | Low (cosmetic spoof) | Change another widget's popout title. UX-only.                                                                            |

### ⚠️ Native-dialog hardening for `widget-mcp:set-grant`

The `widget-mcp:set-grant` handler now runs a diff (`grantDiff.js`)
between the widget's current grant and the proposed change. If the
new grant **broadens** permissions in any dimension (new server, new
tool, new path, new host, `*` wildcard) the handler calls
`dialog.showMessageBox` — Electron's OS-native modal — listing the
specific additions and asking the user to confirm. Cancel drops the
write; Allow proceeds.

**Why native:** the prompt is a system-level dialog; a renderer
(including a malicious widget in the same JS realm) cannot dismiss it
programmatically or synthesize clicks against it. This closes the
previous consent-bypass exploit:

```js
// Was a silent total-bypass before this slice; now triggers an
// OS-native "Allow new permissions for @malicious/self?" dialog.
mainApi.widgetMcp.setGrant("@malicious/self", {
    servers: { fs: { tools: ["*"], readPaths: ["/"], writePaths: ["/"] } },
    domains: { network: { hosts: ["*"] } },
});
```

**Reductions / equality** (revocations, partial trims, no-ops) pass
unprompted — the user only sees a confirm when permissions are
actually being added.

**Known limitations** (carried over from the broader trust-model
caveat — full fix is multi-week per-widget BrowserView refactor):

-   `widget-mcp:revoke` and `widget-mcp:revoke-server` still don't
    confirm. A malicious widget can DoS another widget's grant. Lower
    severity than the bypass; not addressed in this slice.
-   The data-plane widgetId-spoofing remains: a widget can still
    claim to be another widget at gate-call time. The native-dialog
    fix is specifically for grant-writing (control plane). The runtime
    gate stack continues to trust renderer-supplied widgetId.
-   A malicious widget could DoS the user via repeated `setGrant`
    calls spamming dialogs. The user will notice immediately and
    uninstall the widget.

The fix requires either:

1. **WidgetId trust-model tightening** — bind `widgetId` to the
   caller's `webContents.id` via a registry, and reject IPC calls
   whose claimed widgetId doesn't match. Closes both this and the
   existing widgetId-spoofing caveat at once. Multi-day slice.
2. **Preload split** — move `widgetMcp.*` (and other admin channels)
   off `mainApi` and onto a separate bridge only the Settings panel /
   consent modal preload exposes. Requires distinct preloads per
   BrowserWindow context; possible but invasive.

Until one of those ships, the gap is exploitable. **The
trust-model slice is now the priority next move.**

### Heuristic limitations (known, accepted)

-   The delegate-extractor takes the first identifier-shaped call after
    `getSenderWindow(e)`. Multi-step dispatchers (e.g. a switch that
    routes by message type) won't be picked up; those handlers will
    classify as `system` even when the routed-to function gates. Fix
    requires real AST traversal — out of scope for this slice.
-   A "gated" classification requires the delegate function name to
    appear in a controller file with a gate ref within ~80 lines of its
    declaration. Refactors that move the gate into a helper called more
    than 80 lines later will silently downgrade the classification.
-   Inline string-literal channels are now detected (the previous
    "intentionally skipped" limitation is resolved as of the parser
    improvement). Both single- and double-quoted forms are accepted in
    `ipcRenderer.invoke()` and `ipcMain.handle()` / `loggedHandle()`.
-   Constant resolution uses a simple `const NAME = "value"` regex
    against `dash-core/electron/events/*Events.js`. Computed values
    (`const FOO = PREFIX + "bar"`) are NOT resolved; if a future
    events file introduces them, the audit will treat the constant as
    unresolved (leading to a false `phantom`/`dead` if both sides
    don't use the same form).
